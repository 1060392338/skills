# 工作流协议 v1

## 总则

用户发需求 → translator（需求翻译）→ 用户确认需求规格 → 我出完整剧本 → 用户审查确认 → 按剧本执行

禁止：不经过规划直接开发。禁止：跳过审查步骤。

---

## 剧本结构（完整模板）

```
需求：xxx

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
阶段一：调研与规划
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目标：输出可执行的开发方案

步骤 0 — 需求翻译
  - 角色：[translator]
  - 输入：用户原始需求
  - 产出：需求规格说明书（业务目标、功能清单、约束、假设、风险）
  - 用户确认后才进入步骤 1.1
   → 交付物传给步骤 1.1

步骤 1.1 — 情报调研
  - 角色：[translator 产出 + product-trend-researcher / web_search]
  - 产出：调研报告（当前技术栈、竞品方案、最新对抗手段）
  - 耗时估计：xx 分钟
   → 交付物传给步骤 1.2

步骤 1.2 — 架构方案
  - 角色：[engineering-software-architect]
  - 输入：步骤 1.1 调研报告
  - 产出：技术方案文档（含架构图概要、技术选型、风险清单、备选方案）
  - 风险清单格式：
    - [P0] 致命风险（无法绕过 → 熔断，通知用户决策）
    - [P1] 高风险（有应对方案但代价大）
    - [P2] 中低风险（有常规方案）
  - 耗时估计：xx 分钟
   → 交付物传给步骤 1.3

步骤 1.3 — 方案审查
  - 角色：[engineering-code-reviewer]
  - 输入：步骤 1.2 方案文档
  - 审查维度：
    a) 架构合理性：方案是否过度设计 / 过于简陋
    b) 风险覆盖：风险清单是否完整，每项是否都有应对
    c) 可行性：以当前技术栈能否实现
    d) 可维护性：后续迭代是否困难
  - 标记规则：
    - Blocking ⛔ → 方案不可接受，暂停流程，通知用户决策
    - Warning ⚠️ → 有问题但可改进，附带修改建议
    - Approval ✅ → 方案可执行
  - 产出：审查报告 + 修改建议
   → 交付物传给步骤 1.4

步骤 1.4 — 任务拆分
  - 角色：[project-manager-senior]
  - 输入：步骤 1.3 通过的方案 + 审查意见
  - 产出：按顺序的步骤清单，每步包含：
    a) 步骤编号和名称
    b) 负责角色（1 个主写 + 几个辅助）
    c) 输入条件（依赖上一步的什么产出）
    d) 交付物（代码/配置/文档）
    e) 测试要求
    f) 预估耗时
    g) 已知风险

[用户审查阶段一全部产出 → 确认或要求修改 → 确认后进入阶段二]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
阶段二：逐步骤开发
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

每个步骤走循环：

步骤 N：xxx
├── 主开发角色：[1 个主写角色]
├── 辅助角色：[可选，如 database-optimizer 参与数据层]
├── 输入：[上一步交付物 / 阶段一方案]
├── 子流程：
│   ├── 1. 主开发角色写代码
│   ├── 2. 主开发角色写单元测试（同一人）
│   ├── 3. 主开发角色验证：测试通过 + 代码可运行
│   ├── 4. engineering-code-reviewer 审查
│   │     ├── Blocking ⛔ → 退回修改，重新走步骤循环
│   │     ├── Warning ⚠️ + Approval ✅ → 附带修改意见，进入确认
│   │     └── Approval ✅ → 进入确认
│   └── 5. 用户确认 → 进下一步
└── 交付物：[代码文件列表 + 测试文件列表]
└── 风险：[本步骤已知风险]
└── 熔断条件：审查 Blocking 或步骤内发现 P0 风险

[用户确认本步骤 → 进下一步 / 用户要求修改 → 重新走本步骤循环]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
阶段三：集成与验收
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

步骤 N+1 — 全量代码审查
  - 角色：[engineering-code-reviewer]
  - 范围：全部新增/修改代码
  - 审查维度：同步骤 1.3 + 各步骤之间衔接是否完整
  - 产出：全量审查报告

步骤 N+2 — 性能与安全审查
  - 角色：[engineering-database-optimizer + engineering-software-architect]
  - 范围：数据层设计、安全风险、性能瓶颈
  - 产出：性能/安全报告

步骤 N+3 — 部署方案
  - 角色：[engineering-devops-automator]
  - 产出：部署脚本 / 运行说明
  - 条件：需要部署环境时才执行

[用户最终验收]

交付物：完整代码 + 测试 + 文档 + 运行说明
```

## 横向规则

### 1. 角色调用规则

| 步骤类型 | 默认角色 | 可替换/补充 |
|---|---|---|
| 需求翻译 | translator | 小需求可由我自己直接翻译 |
| 情报调研 | product-trend-researcher | 直接用 web_search 替代（小需求不额外调 agent） |
| 架构设计 | engineering-software-architect | 简单需求可合并到 senior-developer |
| 代码审查 | engineering-code-reviewer | 所有步骤固定角色，不变 |
| 前端开发 | engineering-frontend-developer | - |
| 后端开发 | engineering-senior-developer | 复杂逻辑可加 senior-developer + software-architect |
| 数据库/数据层 | engineering-database-optimizer | - |
| 原型验证 | engineering-rapid-prototyper | 存在不确定的技术方案时启用 |
| DevOps/部署 | engineering-devops-automator | 仅需要部署时启用 |
| Git 工作流 | engineering-git-workflow-master | 分支策略 / 合并冲突解决 |
| 任务拆分/排期 | project-manager-senior | 少于 3 个步骤可不调，我手动拆分 |
| 产品方案 | product-manager | 涉及用户体验/功能定义时启用 |
| 排期优先级 | product-sprint-prioritizer | 多需求并行时启用 |
| 用户行为/增长 | product-behavioral-nudge-engine | 涉及留存/转化策略时启用 |
| 反馈分析 | product-feedback-synthesizer | 已有用户反馈需要分析时启用 |
| 趋势研究 | product-trend-researcher | 同情报调研 |
| 项目协调 | project-management-project-shepherd | 涉及多个团队/跨周的项目 |
| 架构原则 | engineering-backend-architect | 有 senior-developer 时可替代 software-architect |
| Git 规范 | engineering-git-workflow-master | 仅多人协作项目需要 |

**原则：** 能少调就少调，不浪费响应时间。3 步以内的小需求可以只调 1 个代码角色 + code-reviewer。

### 2. 熔断机制

- **code-reviewer 标记 Blocking ⛔** → 自动暂停当前流程
  - 属于阶段一方案审查 → 通知用户决策（放弃/修改/绕行）
  - 属于阶段二代码审查 → 退回给开发角色修改，重新走步骤循环
  - 若同一步骤被 Blocking 退回超过 2 次 → 升级通知用户决策

- **开发中发现 P0 风险**（如"反爬完全无法绕过"）→ 暂停，通知用户

- **用户可随时叫停**：在任意步骤说"停"、"不做了"、"换个方案" → 记录当前进度，停止执行

### 3. 步骤验收标准（默认）

每步骤完成后，必须同时满足以下条件才能进入确认：

1. ✅ 代码可运行（不报错退出）
2. ✅ 该步骤的单元测试全部通过
3. ✅ code-reviewer 无 Blocking 意见
4. ✅ 没有新增 P0/P1 风险

如果 code-reviewer 有 Warning ⚠️ 但标记了 Approval ✅，附带修改意见但不阻塞流程。

### 4. 单步骤重试限制

同一步骤被 code-reviewer 退回修改，最多 2 次。
- 第 1 次退回 → 正常修改重走
- 第 2 次退回 → 通知用户"这个步骤反复被拒，建议你介入看看"

### 5. 子 agent 并发限制

考虑到当前配置 `maxConcurrent: 2`，sub-agents `maxConcurrent: 8`：

- 阶段一：串行执行，一次只调 1 个角色（上一个交付了再调下一个）
- 阶段二：每步骤串行，但不禁止主开发角色内部并行子任务
- 如果用户觉得太慢，可以告诉我"加快"，我会评估并行风险

### 6. 单元测试归属

- 统一规则：**谁写代码，谁写单元测试**
- code-reviewer 审查时同时审查代码和测试
- 测试覆盖要求：
  - 核心逻辑（采集/解析/计算）：分支覆盖率 ≥ 80%
  - 工具类/辅助函数：不强制，酌情
  - 禁止：测试只测 happy path 不覆盖异常

## 快速通道（小需求例外）

以下情况可跳过阶段一的调研和架构步骤，直接从阶段二开始：

- **需求 ≤ 3 步即可完成**
- **用户明确说"直接写，不用规划"**
- **纯工具类/辅助函数开发**

跳过时我仍会出一个**简单步骤清单**，确保你不会迷失。

---

## 交付物清单（最终）

1. 所有代码文件
2. 所有测试文件
3. 运行说明（README 或直接告知）
4. 风险报告（已知风险 + 未处理风险 + 后续建议）
