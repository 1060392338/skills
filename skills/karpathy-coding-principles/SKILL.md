# Karpathy AI编程原则

## 概述

源自 Andrej Karpathy 从实际观察中总结的 LLM 编码陷阱，93K+ GitHub stars（forrestchang/andrej-karpathy-skills）。

**Karpathy 核心洞察：**
> "模型会代你做错误假设，不假思索执行，不管理困惑，不寻求澄清，不呈现矛盾/权衡，该反驳时也不反驳。"

> "它们喜欢把代码和 API 搞复杂，堆砌抽象，不清理死代码……100行能搞定非写成1000行。"

**这些原则偏向谨慎而非速度。** 对于简单任务需要自行判断。

## 四大原则

### 1. Think Before Coding（编码前思考）

**核心：不要假设，不要隐藏困惑，呈现权衡**

在实现之前：
- **明确说明假设** — 如果不确定，询问而不是猜测
- **呈现多种解释** — 当存在歧义时，不要默默选择
- **适时提出异议** — 如果存在更简单的方法，说出来
- **困惑时停下来** — 指出不清楚的地方并要求澄清

**反模式：** 用户说"添加导出功能"，LLM 默默假设导出所有用户、文件路径、字段名，不问。

### 2. Simplicity First（简洁优先）

**核心：最少的代码解决问题，不做推测**

- 不添加要求之外的功能
- 不为一次性代码创建抽象
- 不添加未要求的"灵活性"或"可配置性"
- 不为不可能发生的场景做错误处理
- 如果 200 行可以写成 50 行，重写它

**自检问题：** 资深工程师会觉得这过于复杂吗？如果是，简化。

**反模式：** 一个简单的折扣计算函数被实现成策略模式、抽象类、配置类的 30+ 行架构。

### 3. Surgical Changes（精准修改）

**核心：只碰必须碰的，只清理自己造成的混乱**

编辑现有代码时：
- **不要"改进"** 相邻的代码、注释或格式
- **不要重构** 没坏的东西
- **匹配现有风格**，即使你更倾向于不同的写法
- 如果注意到无关的死代码，**提一下——不要删除它**

当你的改动产生孤儿代码时：
- 删除因你的改动而变得无用的导入/变量/函数
- 不要删除预先存在的死代码，除非被要求

**检验标准：** 每一行修改都应该能直接追溯到用户的请求。

**反模式：** 修复一个 bug 时顺便添加了类型提示、改了引号风格、添加了文档字符串。

### 4. Goal-Driven Execution（目标驱动执行）

**核心：定义成功标准。循环验证直到达成。**

将指令式任务转化为可验证目标：

| 不要这样做... | 转化为... |
|--------------|-----------------|
| "添加验证" | "为无效输入编写测试，然后让它们通过" |
| "修复 bug" | "编写重现 bug 的测试，然后让它通过" |
| "重构 X" | "确保重构前后测试都能通过" |

对于多步骤任务，说明一个简短的计划：
```
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]
```

**反模式：** "我会修复认证系统：1. review 代码 2. 找出问题 3. 做改进 4. 测试" —— 没有具体成功标准。

## 何时使用

**必须在此类场景加载此技能：**
- 用户给出一个模糊的编程任务
- 用户要求"添加功能"、"修复问题"、"改进代码"
- 多步骤实现任务
- 任何涉及修改现有代码的任务

**立即应用的判断：**
- 需求模糊 → 先列假设清单（原则1）
- 准备写超过 20 行代码 → 自问能更简单吗（原则2）
- 修改现有文件 → 检查每一行是否必要（原则3）
- 接到多步骤任务 → 定义可验证的成功标准（原则4）

## 检验标准

这些原则执行良好的标志：
- diff 中不必要的改动变少
- 因过度复杂导致的返工变少
- 澄清问题出现在实施之前而非错误之后
